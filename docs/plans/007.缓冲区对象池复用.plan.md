# 缓冲区对象池复用

在 SOCKS5 代理的双向数据转发（热路径）中，当前每个连接都会分配 2 个 64 KiB 的 `BufReader` 缓冲区，连接结束后释放。在高频短连接场景下，频繁的堆分配和释放会产生不可忽视的开销。通过引入缓冲区对象池，可以复用已分配的缓冲区，避免重复分配。

## Proposed Changes

### 新增 BufferPool 模块

#### [NEW] [buffer_pool.rs](file:///Users/jin.gjm/code/rust/exsocks/src/buffer_pool.rs)

新建 `buffer_pool.rs` 模块，实现基于 `crossbeam::ArrayQueue` 的无锁缓冲区对象池：

- **`BufferPool` 结构体**：持有一个 `ArrayQueue<Vec<u8>>` 作为空闲缓冲区队列
  - `new(capacity: usize, buffer_size: usize)` — 创建池，`capacity` 为池中最大缓冲区数量，`buffer_size` 为每个缓冲区的字节大小
  - `get(&self) -> Vec<u8>` — 从池中取出一个缓冲区，池空时分配新的 `Vec<u8>` 并预分配 `buffer_size` 容量
  - `put(&self, buf: Vec<u8>)` — 归还缓冲区到池中，池满时直接丢弃（由系统回收）
  - `buffer_size(&self) -> usize` — 返回缓冲区大小，供外部查询

> [!NOTE]
> **设计决策**：不实现自定义的 `PooledBufReader`（即不包装 `AsyncBufRead` trait），而是让 `relay` 函数从池中取出 `Vec<u8>`，手动构建标准的 `tokio::io::BufReader`，拷贝完成后手动归还。原因：
> 1. `BufReader` 内部会 take ownership 底层缓冲区，自定义 `AsyncBufRead` 实现复杂且容易出错
> 2. 手动 get/put 模式更简单直接，且 `relay` 函数是唯一使用点
> 3. 避免引入不必要的抽象层

> [!TIP]
> 选择 `crossbeam-queue` 的 `ArrayQueue` 而非 `tokio::sync::Mutex<Vec>` 的原因：
> - `ArrayQueue` 是无锁的，`push`/`pop` 仅需一次 CAS 操作
> - 在高并发场景下不会产生锁竞争
> - 固定容量，内存占用可预测

### Relay 模块集成

#### [MODIFY] [relay.rs](file:///Users/jin.gjm/code/rust/exsocks/src/relay.rs)

修改 `relay` 函数，使用 `BufferPool` 提供的缓冲区：

- `relay` 函数签名变更：`buffer_size: usize` 改为 `pool: &Arc<BufferPool>`
- 在 `relay` 入口从池中 `get()` 两个缓冲区
- 将 `Vec<u8>` 传入 `copy_with_shutdown`，内部用它构建 `BufReader`
- 拷贝完成后，手动 `put()` 归还缓冲区

`copy_with_shutdown` 签名调整为接收预分配的 `Vec<u8>` 缓冲区，并在完成后返回该缓冲区以便归还：

```diff
- async fn copy_with_shutdown<R, W>(
-     reader: &mut R, writer: &mut W, buffer_size: usize
- ) -> io::Result<u64>
+ async fn copy_with_shutdown<R, W>(
+     reader: &mut R, writer: &mut W, buf: Vec<u8>
+ ) -> io::Result<(u64, Vec<u8>)>
```

```diff
  pub async fn relay(
      client: TcpStream,
      target: TcpStream,
-     buffer_size: usize,
+     pool: &Arc<BufferPool>,
      cancel: CancellationToken,
  ) -> Result<(u64, u64), SocksError> {
```

> [!IMPORTANT]
> 由于 `copy_with_shutdown` 在 `tokio::spawn` 内执行，缓冲区需要 move 进 spawn 闭包，完成后通过返回值传回。归还操作在 `relay` 函数中 `tokio::join!` 收集结果后执行，确保无论成功还是失败都能归还缓冲区。
