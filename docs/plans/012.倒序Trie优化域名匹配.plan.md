# 倒序 Trie 优化域名后缀匹配

## 背景

在 `TargetRuleSet` 的域名匹配实现中，DOMAIN-SUFFIX 原先使用 `HashMap<String, Vec<PrioritizedRule>>` 存储后缀规则，查找时需要逐级拆分域名并多次 hash 查找。

以 `api.test.com.cn` 为例，原方案需要：
- 1 次 `domain_index` HashMap 查找（精确匹配）
- 4 次 `suffix_index` HashMap 查找（逐级后缀：`api.test.com.cn` → `test.com.cn` → `com.cn` → `cn`）
- hash 总字节数约 52 字节（完整子串累加）
- 无法提前终止

## 方案

将 `suffix_index: HashMap<String, Vec<PrioritizedRule>>` 替换为倒序域名 Trie（Reverse Domain Trie）。

### 核心思想

域名的层级结构天然是从右到左的（`cn` → `com.cn` → `test.com.cn`），与 Trie 的前缀匹配方向一致。将域名按 `.` 分割后**倒序插入** Trie，就能用一次遍历同时完成所有后缀匹配。

### 数据结构

```rust
struct TrieNode {
    children: HashMap<Box<str>, TrieNode>,   // label → child
    suffix_rules: Vec<PrioritizedRule>,       // DOMAIN-SUFFIX 规则
}
```

Trie 结构示例（规则: DOMAIN-SUFFIX baidu.com, DOMAIN-SUFFIX test.com.cn）：

```text
Root
 ├─ "com"
 │   └─ "baidu" [suffix_rules: ...]
 └─ "cn"
     └─ "com"
         └─ "test" [suffix_rules: ...]
```

### 查找过程（以 `api.test.com.cn` 为例）

```
输入: "api.test.com.cn"
分割并倒序: ["cn", "com", "test", "api"]

遍历 Trie:
  Step 1: root.children.get("cn")   → hash 2B, 找到节点, 检查 suffix_rules
  Step 2: node.children.get("com")  → hash 3B, 找到节点, 检查 suffix_rules
  Step 3: node.children.get("test") → hash 4B, 找到节点, 检查 suffix_rules ✓ 命中（继续遍历，更深层可能有更高优先级规则）
  Step 4: node.children.get("api")  → hash 3B, 无子节点, break 终止
```

### 性能对比

| 维度 | 旧方案（HashMap 逐级查找） | 新方案（倒序 Trie） |
|------|---------------------------|---------------------|
| Hash 次数（suffix 部分） | L 次（L=域名层级数） | 最多 L 次 |
| Hash 输入长度 | 完整子串（52B 总计） | 单个 label（12B 总计） |
| 无效查找 | 大量（如 `cn`、`com.cn` 通常无规则，但仍需 hash 查找） | 无（路径不存在直接终止） |
| 路径断裂时提前终止 | 不支持（必须遍历所有层级） | 支持（子节点不存在即 break） |

**核心收益**：旧方案必须对每一级后缀都做一次完整的 HashMap 查找，即使绝大多数后缀（如 `cn`、`com.cn`）在规则库中根本不存在，查找仍然不可跳过。新方案中，Trie 路径不存在时直接终止遍历，避免了这些无效查找。hash 输入长度从完整子串缩短为单个 label 是附带收益，在现代 CPU 上实际性能差异很小。

**注意**：这里的"提前终止"指的是 Trie 路径断裂时停止遍历（子节点不存在即 break），而**不是**命中规则后停止。命中规则后仍需继续遍历剩余 labels，因为更深层节点可能存在优先级更高（priority 更小）的规则。

### 设计决策

1. **DOMAIN 精确匹配保留 HashMap**：精确匹配本身就是 O(1) 单次 hash，无需优化
2. **使用 `Box<str>` 作为 Trie key**：比 `String` 少 8 字节内存开销
3. **优先级处理不变**：Trie 遍历路径上收集所有命中规则，取全局 priority 最小的
4. **使用 `SmallVec<[&str; 8]>` 替代 `Vec<&str>` 收集倒序 labels**：域名层级通常 3-5 级，远小于容量 8，完全在栈上分配，零堆分配。Benchmark 数据表明平均提升 22-24%，深层域名和未命中场景提升 30-39%（详见 `docs/tests/domain-match-vec-vs-smallvec-benchmark.md`）
