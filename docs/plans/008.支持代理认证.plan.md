### socks5-auth-support ###
为 exsocks SOCKS5 代理服务器添加 RFC1929 用户名/密码认证支持，包括：通过 server.yaml 配置是否启用认证、从 user.yaml 读取用户凭证列表、使用 notify crate 实现 user.yaml 文件变更自动热加载。未启用认证时同时接受无认证和用户名密码认证（不校验凭证）。


# SOCKS5 用户名/密码认证支持

为 exsocks 添加 RFC1929 用户名/密码认证功能。支持两种运行模式：

- **认证模式**（`auth_enabled: true`）：强制要求客户端使用用户名密码认证（METHOD 0x02），从 `user.yaml` 加载凭证并严格校验
- **非认证模式**（`auth_enabled: false`，默认）：同时接受无认证（0x00）和用户名密码认证（0x02）两种方式，但不实际校验凭证，全部视为通过

## User Review Required

> [!IMPORTANT]
> 密码目前以明文存储在 `user.yaml` 中。如果后续需要支持哈希存储（如 bcrypt/argon2），可作为独立迭代实现。

> [!WARNING]
> 启用认证后，所有不支持用户名密码认证的客户端将被拒绝连接。请确保客户端配置了正确的认证信息。

### 用户认证存储模块（新增）

#### [NEW] [auth.rs](file:///Users/jin.gjm/code/rust/exsocks/src/auth.rs)

新增用户认证存储模块，负责：
1. 定义 `UserEntry` 结构（username + password）
2. 定义 `UserStore` 结构，使用 `ArcSwap<HashMap<String, String>>` 实现无锁读、原子替换的用户凭证存储
3. 提供 `load_from_file(path)` 方法从 YAML 文件加载用户列表
4. 提供 `verify(username, password) -> bool` 方法验证凭证
5. 提供 `reload()` 方法重新加载文件并原子替换内部 HashMap
6. 启动文件监听任务 `watch()` 使用 `notify` crate 监听 user.yaml 变更，变更时自动调用 `reload()`

核心数据结构设计：

```rust
/// user.yaml 的反序列化结构
#[derive(Debug, Deserialize)]
struct UserConfig {
    users: Vec<UserEntry>,
}

#[derive(Debug, Deserialize)]
struct UserEntry {
    username: String,
    password: String,
}

/// 线程安全的用户凭证存储，支持热加载
pub struct UserStore {
    /// 用户名 -> 密码的映射，使用 ArcSwap 实现无锁读 + 原子替换
    credentials: ArcSwap<HashMap<String, String>>,
    /// 配置文件路径
    path: PathBuf,
}
```

> [!TIP]
> 使用 `ArcSwap` 而非 `RwLock` 的原因：每次认证请求都需要读取用户表，`ArcSwap::load()` 是无锁操作（仅一次原子 load），比 `RwLock::read()` 更高效。热加载时通过 `store()` 原子替换整个 HashMap，无需持锁。

文件监听设计：
- 使用 `notify::RecommendedWatcher` 监听 user.yaml 所在目录
- 过滤 `ModifyKind::Data` 和 `CreateKind::File` 事件
- 添加 500ms 防抖，避免编辑器保存时触发多次重载
- 重载失败时仅打印错误日志，保留旧的用户表不受影响

#### [MODIFY] [handshake.rs](file:///Users/jin.gjm/code/rust/exsocks/src/socks5/handshake.rs)

重构握手函数，支持两种运行模式：

```rust
/// 执行 SOCKS5 握手。
///
/// - 当 `user_store` 为 `Some` 时（认证模式）：
///   强制要求客户端使用用户名密码认证（METHOD 0x02），严格校验凭证
///
/// - 当 `user_store` 为 `None` 时（非认证模式）：
///   同时接受无认证（0x00）和用户名密码认证（0x02）
///   如果客户端选择用户名密码认证，走完子协商流程但不校验凭证，直接返回成功
///   优先选择无认证方式（0x00），仅当客户端不支持 0x00 但支持 0x02 时才走子协商
pub async fn perform_handshake(
    stream: &mut TcpStream,
    user_store: Option<&UserStore>,
) -> Result<(), SocksError>
```

**非认证模式的方法选择逻辑：**
1. 客户端支持 0x00 → 选择无认证，直接通过
2. 客户端不支持 0x00 但支持 0x02 → 选择用户名密码认证，走子协商但跳过校验（直接回复成功）
3. 客户端既不支持 0x00 也不支持 0x02 → 回复 0xFF，拒绝连接

**认证模式的方法选择逻辑：**
1. 客户端支持 0x02 → 选择用户名密码认证，走子协商并严格校验
2. 客户端不支持 0x02 → 回复 0xFF，拒绝连接

RFC1929 子协商流程（新增私有函数 `perform_username_password_auth`）：
1. 读取 `[VER, ULEN]`，校验 VER == 0x01
2. 读取 ULEN 字节的用户名
3. 读取 `[PLEN]`
4. 读取 PLEN 字节的密码
5. 如果 `user_store` 为 `Some`，调用 `user_store.verify(username, password)` 校验
6. 如果 `user_store` 为 `None`，跳过校验，直接视为通过
7. 成功回复 `[0x01, 0x00]`，失败回复 `[0x01, 0x01]` 并返回错误
