### dns-cache ###
为 DNS 解析正常默认缓存300秒，增加"负缓存"（negative caching）机制：DNS 解析失败时缓存失败结果 30 秒，避免短时间内对同一不可达域名反复发起 DNS 查询。
- **解析成功**：缓存 300 秒（`dns_cache_ttl`，已有）
- **解析失败**：缓存 30 秒（`dns_cache_negative_ttl`，新增），在此期间直接返回缓存的错误，不再重复查询

## 设计要点

- **缓存纯 IP 地址（`Vec<IpAddr>`）而非 `SocketAddr`**：端口不参与缓存 key，连接时动态组合端口，避免同一域名不同端口请求导致缓存失效
- **`DnsCache` 只负责 DNS 解析和缓存，不涉及 TCP 连接**：`resolve()` 返回 `Vec<IpAddr>` 地址列表，调用方（`Address::connect()`）拿到 IP 列表后自行建立连接、控制超时。职责分离使 DNS 缓存可独立测试，且避免双重超时问题
- **DNS 解析返回空结果视为失败**：`lookup_host` 成功但返回空地址列表时，走负缓存路径
- **集成 Prometheus Metrics 监控**：记录缓存命中/未命中、DNS 解析成功/失败、当前缓存条目数等指标
- **并发请求合并（Request Coalescing）**：同一域名同时只有一个 leader 任务执行 DNS 解析，其余 waiter 等待通知后从缓存获取结果，避免 thundering herd 问题。使用 `InFlightGuard` drop guard 确保 future 被取消时 in_flight 条目能正确清理
- **Per-IP 连接超时由 `Address::connect()` 控制**：`try_connect()` 对每个 IP 地址的 `TcpStream::connect` 使用独立的 `connect_timeout`，防止单个不可达 IP 占用系统默认 TCP 超时（75-120 秒），确保多个 IP 地址都有机会被尝试。外层 `server.rs` 的 `timeout()` 作为总体兜底
- **可选自定义 DNS 解析服务器**：通过 `dns_resolve_server` 配置项指定自定义 DNS 服务器地址（如 `8.8.8.8`），空字符串或不配置时使用系统默认 DNS 解析器。使用 `hickory-resolver` 库实现自定义 DNS 服务器支持，构建时禁用其内置缓存（`cache_size = 0`），由 `DnsCache` 统一管理缓存

新增 `InFlightGuard` drop guard，确保 in_flight 条目在 resolve 完成或被取消时都能正确清理：

```rust
struct InFlightGuard<'a> {
    in_flight: &'a DashMap<String, Arc<Notify>>,
    domain: String,
    notify: Arc<Notify>,
}

impl Drop for InFlightGuard<'_> {
    fn drop(&mut self) {
        self.in_flight.remove(&self.domain);
        self.notify.notify_waiters();
    }
}
```


新增 `build_custom_resolver()` 静态方法：根据 `dns_server` 地址构建 `hickory-resolver` 的 `TokioAsyncResolver`，配置 UDP+TCP 双协议、禁用内置缓存、设置超时和重试次数。无效地址时 warn 并回退到系统默认。

4. **`resolve()` 方法重新设计**（返回 `Vec<IpAddr>`，纯 DNS 解析 + 并发请求合并）：
   - 签名：`pub async fn resolve(&self, domain: &str) -> Result<Vec<IpAddr>, SocksError>`
   - 拆分为三个内部方法：`resolve()`（入口 + 并发合并）、`check_cache()`（缓存检查）、`do_resolve()`（实际解析）
   - **并发请求合并**：使用 `DashMap::entry()` 原子性判断 in_flight 状态
     - `Entry::Vacant` → 成为 leader，创建 `Notify` 插入 in_flight，通过 `InFlightGuard` 确保清理
     - `Entry::Occupied` → 成为 waiter，先注册 `notified()` future 再释放锁，等待 leader 完成后从缓存获取结果
     - waiter 被通知后缓存仍为空（leader 被取消）→ loop 重试成为新 leader
   - **`check_cache()`**（同步方法）：根据 `DnsResult` 类型选择对应 TTL 检查，过期则移除返回 None；正缓存命中返回 `Ok(Vec<IpAddr>)`，负缓存命中返回错误
   - **`do_resolve()`**：根据 `custom_resolver` 是否存在选择解析方式，分发到 `resolve_with_system()` 或 `resolve_with_custom()`
   - **`resolve_with_system()`**：使用系统 `lookup_host` 解析（原有逻辑）
   - **`resolve_with_custom()`**：使用 `hickory-resolver` 的 `lookup_ip()` 通过自定义 DNS 服务器解析
   - 两种解析方式统一行为：
     - 解析成功但返回空地址列表 → 视为失败，缓存负结果
     - 解析成功且有地址 → 直接缓存 IP 地址列表，返回给调用方
     - 解析失败 → 缓存负结果（`DnsResult::Failure`），返回 `SocksError::InvalidAddress`
     - 超时 → 缓存负结果，返回超时错误
   - 容量检查：在缓存未命中且 `cache.len() >= max_entries` 时，先调用 `evict_expired()` 清理

5. **`evict_expired()` 增强**：根据条目类型使用对应的 TTL 判断是否过期

6. **Metrics 监控埋点**（依赖 `metrics_registry.rs` 中的常量）：
   - 缓存命中时：`counter!(DNS_CACHE_TOTAL, "result" => "hit").increment(1)`
   - 缓存未命中时：`counter!(DNS_CACHE_TOTAL, "result" => "miss").increment(1)`
   - DNS 解析成功时：`counter!(DNS_RESOLVE_TOTAL, "result" => "success").increment(1)`
   - DNS 解析失败时：`counter!(DNS_RESOLVE_TOTAL, "result" => "failure").increment(1)`
   - 缓存条目变更时：`gauge!(DNS_CACHE_ENTRIES).set(self.cache.len() as f64)`

### SOCKS5 协议层适配

#### [MODIFY] [socks5/protocol.rs](file:///Users/jin.gjm/code/rust/exsocks/src/socks5/protocol.rs)

`Address::connect()` 方法增加 `dns_cache` 和 `connect_timeout` 参数，承接 DNS 解析后的 TCP 连接职责：

```rust
pub async fn connect(
    &self,
    port: u16,
    dns_cache: Option<&DnsCache>,
    connect_timeout: Duration,
) -> Result<TcpStream, SocksError> {
    match self {
        Address::IPv4(addr) => { /* 带超时直接连接 SocketAddr */ }
        Address::IPv6(addr) => { /* 带超时直接连接 SocketAddr */ }
        Address::Domain(domain) => {
            if let Some(cache) = dns_cache {
                let addrs = cache.resolve(domain).await?;
                Self::try_connect(&addrs, port, connect_timeout).await
            } else {
                Ok(TcpStream::connect((domain.as_str(), port)).await?)
            }
        }
    }
}
```

新增两个辅助方法：

- **`connect_with_timeout()`**：对单个 `SocketAddr` 执行带超时的 TCP 连接，IPv4/IPv6 直连场景统一使用
- **`try_connect()`**（从 `dns_cache.rs` 迁移）：依次尝试 IP 地址列表中的每个 IP，每个 IP 有独立的 `connect_timeout`，返回第一个成功的 `TcpStream`
