### 添加Prometheus-Metrics监控 ###
为 exsocks SOCKS5 代理服务器添加基于 metrics + metrics-exporter-prometheus 的 Prometheus 监控指标，使用 hyper 暴露轻量级 /metrics HTTP 端点，仅使用 Counter 和 Gauge 类型（无 Histogram），共 9 个指标。

# 为 exsocks 添加 Prometheus Metrics 监控

## 背景

exsocks 是一个高性能异步 SOCKS5 代理服务器，当前仅通过 `tracing` 日志记录运行状态，缺少结构化的监控指标。本方案使用 `metrics` + `metrics-exporter-prometheus` crate 添加 Prometheus 兼容的监控指标，并通过 `hyper` 暴露轻量级 HTTP 端点供 Prometheus 抓取。

**设计原则：仅使用 Counter 和 Gauge，不使用 Histogram，避免直方图桶计算开销，保持极致精简。**

## User Review Required

> [!IMPORTANT]
> metrics HTTP 端点将在独立端口监听（默认 `127.0.0.1:9090`），需确认该端口在部署环境中未被占用。可通过配置 `metrics_bind` 修改，或设置 `metrics_enabled: false` 完全禁用。

### Metrics 核心模块（新建）

#### [NEW] [metrics_registry.rs](file:///Users/jin.gjm/code/rust/exsocks/src/metrics_registry.rs)

集中定义所有 metrics 指标名称常量和描述注册。所有指标通过 `metrics` crate 的宏（`counter!`、`gauge!`）在使用处按需记录。

**指标清单（共 9 个，仅 Counter + Gauge）：**

| 类型 | 指标名 | 标签 | 说明 |
|------|--------|------|------|
| Gauge | `exsocks_active_connections` | - | 当前活跃连接数 |
| Counter | `exsocks_connections_total` | `status`=accepted/blocked | 连接总数（含白名单拒绝） |
| Counter | `exsocks_bytes_total` | `direction`=up/down | 传输字节总数 |
| Counter | `exsocks_connect_target_errors_total` | - | 连接目标失败总数 |
| Counter | `exsocks_auth_total` | `result`=success/failure | 认证结果计数 |
| Counter | `exsocks_dns_cache_total` | `result`=hit/miss | DNS 缓存命中/未命中 |
| Counter | `exsocks_dns_resolve_total` | `result`=success/failure | DNS 解析结果 |
| Counter | `exsocks_target_rule_total` | `action`=pass/block | 目标规则命中计数 |
| Gauge | `exsocks_dns_cache_entries` | - | DNS 缓存当前条目数 |

本模块提供：
1. **指标名称常量定义** - 避免字符串散落各处
2. **`describe_metrics()` 函数** - 在启动时注册所有指标的描述信息
3. **`init_metrics_recorder()` 函数** - 初始化 `PrometheusRecorder` 并返回 `PrometheusHandle`

### Metrics HTTP 端点（新建）

#### [NEW] [metrics_server.rs](file:///Users/jin.gjm/code/rust/exsocks/src/metrics_server.rs)

使用 hyper 1.x 实现最小化 HTTP server，仅响应 `GET /metrics`：

```rust
pub async fn serve_metrics(
    bind: SocketAddr,
    handle: PrometheusHandle,
    cancel: CancellationToken,
) -> Result<(), SocksError> {
    let listener = TcpListener::bind(bind).await?;
    info!("Metrics server listening on http://{}/metrics", bind);
    
    loop {
        tokio::select! {
            result = listener.accept() => {
                let (stream, _) = result?;
                let handle = handle.clone();
                tokio::spawn(async move {
                    // 仅处理 GET /metrics，其他路径返回 404
                    // 返回 handle.render() 的文本内容
                });
            }
            _ = cancel.cancelled() => break,
        }
    }
    Ok(())
}
```

> [!TIP]
> 该 HTTP server 极其轻量，仅在 Prometheus 抓取时（通常 15-30s 间隔）处理请求，对代理性能几乎无影响。

```rust
// 活跃连接计数 guard - 利用 Drop 确保连接关闭时一定 decrement
struct ActiveConnectionGuard;
impl Drop for ActiveConnectionGuard {
    fn drop(&mut self) {
        gauge!(ACTIVE_CONNECTIONS).decrement(1);
    }
}

// 在 handle_connection 入口
gauge!(ACTIVE_CONNECTIONS).increment(1);
let _guard = ActiveConnectionGuard;
```

## 性能影响分析

### 数据路径开销

| 操作 | 开销 | 说明 |
|------|------|------|
| `counter!().increment(1)` | **~5-10ns** | 原子 `fetch_add`，无锁，无分配 |
| `gauge!().increment/decrement` | **~5-10ns** | 同上 |

> [!NOTE]
> 不使用 Histogram，完全避免了直方图桶计算和 `Instant::now()` 系统调用的开销。

### 热路径分析

- **每连接开销**：约 3 次原子操作（active_connections +1/-1、connections_total）≈ **~15-30ns**
- **每次转发完成**：2 次 `counter!` 调用（bytes up/down）≈ **~10-20ns**
- **对比基准**：单次 TCP accept + SOCKS5 握手耗时通常在 **100us-10ms** 级别，metrics 开销占比 < **0.001%**

### 内存开销

- 9 个指标 × ~200 bytes ≈ **~1.8 KB**
- 无 Histogram 桶，无额外内存
- 总额外内存：**< 5 KB**

### metrics_enabled=false 时的开销

不初始化 `PrometheusRecorder`，`metrics` 宏调用走 no-op recorder 路径，开销 **~1-2ns**（仅一次原子 load 检查）。

> [!TIP]
> 总结：metrics 对代理性能的影响可忽略不计。所有指标操作均为无锁原子操作，不引入任何 mutex 竞争或内存分配。
