基于代码评审发现的问题，按优先级分类整理修复方案。

## P0：正确性问题

### Issue 1: Linux splice 实现存在多个致命缺陷

**文件**: `src/relay.rs`

**问题描述**:

当前 Linux 平台的 splice 零拷贝实现存在三个严重问题：

1. **use-after-close 风险**：`relay` 函数接收 `client: TcpStream` 和 `target: TcpStream` 的所有权，但只提取了 `RawFd`（`i32`）传入 `spawn_blocking`。两个 `TcpStream` 在 `relay` 函数的 `tokio::join!` 等待期间可能被 drop（编译器认为它们在提取 fd 后不再被使用），导致阻塞线程中的 fd 指向已关闭的 socket。
2. **`SPLICE_F_NONBLOCK` + `thread::sleep` 轮询是反模式**：在阻塞线程中使用非阻塞 splice，遇到 `EAGAIN` 时 sleep 100μs 重试。这既浪费阻塞线程池资源，又引入最高 100μs 的额外延迟。
3. **pipe fd 泄漏**：`nix::unistd::pipe()` 返回裸 `RawFd`，`splice_loop` 结束后没有显式 `close()`。长时间运行的服务器中，每个连接泄漏 4 个 fd，最终耗尽系统 fd 限制。

**修复方案**:

整体重构 `relay.rs`，采用模块化组织，平台实现隔离到内部 `mod` 中。

核心设计决策：
- **fd 生命周期**：使用 `Arc<std::net::TcpStream>` 持有 socket，确保两个阻塞线程都结束后 fd 才被关闭
- **splice 模式**：读端使用 `SPLICE_F_NONBLOCK`（避免半关闭时一个方向永久阻塞），写端使用阻塞式（确保数据完整写出）。遇到 `EAGAIN` 时用 `thread::yield_now()` 而非 `sleep`，最小化延迟
- **pipe fd 管理**：用 `OwnedFd` 包裹 pipe fd，确保 `splice_loop` 结束时自动关闭
- **返回值**：保留 `(u64, u64)` 字节数统计，用于流量日志

```rust
//! 双向数据转发模块
//!
//! 策略：
//!   - Linux：使用 splice(2) 系统调用实现零拷贝转发。
//!     数据路径：src_socket -> pipe -> dst_socket，全程不经过用户态缓冲区。
//!     由于 splice 是阻塞系统调用，在 tokio::task::spawn_blocking 中执行。
//!   - 非 Linux：使用 tokio::io::copy_bidirectional 作为 fallback。

use tokio::net::TcpStream;
use crate::error::SocksError;

// ─────────────────────────────────────────────────────────────────────────────
// Linux 零拷贝实现（splice）
// ─────────────────────────────────────────────────────────────────────────────

#[cfg(target_os = "linux")]
mod linux_splice {
    use std::os::unix::io::{AsRawFd, FromRawFd, OwnedFd};
    use std::sync::Arc;
    use nix::fcntl::{splice, SpliceFFlags};
    use nix::unistd::pipe;
    use tokio::net::TcpStream;
    use crate::error::SocksError;

    /// 管道缓冲区大小：64 KiB，与 Linux 默认管道容量一致
    const SPLICE_BUF_SIZE: usize = 65536;

    /// 单向 splice 转发：src_fd -> pipe -> dst_fd，直到 EOF。
    /// pipe fd 用 OwnedFd 包裹，函数返回时自动关闭。
    fn splice_loop(src_fd: i32, dst_fd: i32) -> std::io::Result<u64> {
        // 创建 pipe 并用 OwnedFd 管理生命周期
        let (pipe_read_raw, pipe_write_raw) = pipe()
            .map_err(|e| std::io::Error::from_raw_os_error(e as i32))?;
        let pipe_read = unsafe { OwnedFd::from_raw_fd(pipe_read_raw) };
        let pipe_write = unsafe { OwnedFd::from_raw_fd(pipe_write_raw) };

        let pipe_r = pipe_read.as_raw_fd();
        let pipe_w = pipe_write.as_raw_fd();

        let mut total: u64 = 0;

        loop {
            // splice: src_fd -> pipe_w
            // 读端使用 NONBLOCK：当一个方向 EOF 后，另一个方向不会永久阻塞在 splice 上，
            // 而是返回 EAGAIN，配合 yield_now() 等待对端关闭。
            let bytes_read = match splice(
                src_fd, None,
                pipe_w, None,
                SPLICE_BUF_SIZE,
                SpliceFFlags::SPLICE_F_MOVE | SpliceFFlags::SPLICE_F_NONBLOCK,
            ) {
                Ok(0) => break,  // EOF
                Ok(n) => n,
                Err(nix::errno::Errno::EAGAIN) | Err(nix::errno::Errno::EWOULDBLOCK) => {
                    // 非阻塞模式下暂无数据，yield 让出 CPU 后重试
                    std::thread::yield_now();
                    continue;
                }
                Err(e) => return Err(std::io::Error::from_raw_os_error(e as i32)),
            };

            // splice: pipe_r -> dst_fd
            // 写端使用阻塞式（无 NONBLOCK），确保管道中的数据完整写出
            let mut remaining = bytes_read;
            while remaining > 0 {
                match splice(
                    pipe_r, None,
                    dst_fd, None,
                    remaining,
                    SpliceFFlags::SPLICE_F_MOVE,
                ) {
                    Ok(0) => return Ok(total),
                    Ok(written) => remaining -= written,
                    Err(e) => return Err(std::io::Error::from_raw_os_error(e as i32)),
                }
            }
            total += bytes_read as u64;
        }

        // pipe_read 和 pipe_write 在此处自动 close（OwnedFd drop）
        Ok(total)
    }

    /// 基于 splice 的双向零拷贝转发。
    ///
    /// 设计要点：
    /// 1. 用 into_std() 将 tokio TcpStream 转为 std TcpStream
    /// 2. 用 Arc 持有 std TcpStream，确保两个阻塞线程都结束后 fd 才被关闭
    /// 3. 每个方向独立创建 pipe，互不干扰
    pub async fn relay_splice(
        client: TcpStream,
        target: TcpStream,
    ) -> Result<(u64, u64), SocksError> {
        let client_std = client.into_std().map_err(SocksError::Io)?;
        let target_std = target.into_std().map_err(SocksError::Io)?;

        let client_fd = client_std.as_raw_fd();
        let target_fd = target_std.as_raw_fd();

        // Arc 持有 TcpStream，保证 fd 在两个线程都结束后才关闭
        let client_arc = Arc::new(client_std);
        let target_arc = Arc::new(target_std);

        let client_c2t = Arc::clone(&client_arc);
        let target_c2t = Arc::clone(&target_arc);
        let client_t2c = Arc::clone(&client_arc);
        let target_t2c = Arc::clone(&target_arc);

        // 方向1：client -> target
        let c2t_handle = tokio::task::spawn_blocking(move || {
            let _keep_alive = (client_c2t, target_c2t);
            splice_loop(client_fd, target_fd)
        });

        // 方向2：target -> client
        let t2c_handle = tokio::task::spawn_blocking(move || {
            let _keep_alive = (client_t2c, target_t2c);
            splice_loop(target_fd, client_fd)
        });

        let (r1, r2) = tokio::join!(c2t_handle, t2c_handle);

        let client_to_target = r1
            .map_err(|e| SocksError::Io(std::io::Error::other(e)))?
            .map_err(SocksError::Io)?;
        let target_to_client = r2
            .map_err(|e| SocksError::Io(std::io::Error::other(e)))?
            .map_err(SocksError::Io)?;

        Ok((client_to_target, target_to_client))
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// 非 Linux fallback（tokio copy_bidirectional）
// ─────────────────────────────────────────────────────────────────────────────

#[cfg(not(target_os = "linux"))]
mod generic_relay {
    use tokio::io::copy_bidirectional;
    use tokio::net::TcpStream;
    use crate::error::SocksError;

    pub async fn relay_copy(
        mut client: TcpStream,
        mut target: TcpStream,
    ) -> Result<(u64, u64), SocksError> {
        let (c2t, t2c) = copy_bidirectional(&mut client, &mut target).await?;
        Ok((c2t, t2c))
    }
}

// ─────────────────────────────────────────────────────────────────────────────
// 公开接口
// ─────────────────────────────────────────────────────────────────────────────

/// 启动双向数据转发，返回 (client→target 字节数, target→client 字节数)。
pub async fn relay(
    client: TcpStream,
    target: TcpStream,
) -> Result<(u64, u64), SocksError> {
    #[cfg(target_os = "linux")]
    { linux_splice::relay_splice(client, target).await }

    #[cfg(not(target_os = "linux"))]
    { generic_relay::relay_copy(client, target).await }
}
```

**与当前实现的关键差异**:

| 维度 | 当前实现 | 修复后 |
|------|----------|--------|
| fd 生命周期 | 裸 `RawFd`，TcpStream 可能提前 drop | `Arc<std::net::TcpStream>` 保证存活 |
| EAGAIN 处理 | `thread::sleep(100μs)` | `thread::yield_now()` 零延迟 |
| pipe fd 管理 | 裸 `RawFd`，不关闭 | `OwnedFd` 自动关闭 |
| splice 写端 | 也用 `NONBLOCK` | 阻塞式，确保数据完整写出 |
| 代码组织 | 顶层 `#[cfg]` 函数 | 内部 `mod` 隔离平台实现 |
| 返回值 | `(u64, u64)` ✅ | `(u64, u64)` ✅ 保留 |

**关于 `NONBLOCK + yield` 的设计权衡说明**:

读端 splice 使用 `SPLICE_F_NONBLOCK` 而非纯阻塞，是为了解决**半关闭（half-close）问题**：
当一个方向 EOF 时（如客户端关闭了发送端），另一个方向的阻塞 splice 会永久挂起，
因为 `Arc` 持有 fd，fd 不会被关闭来触发唤醒。`NONBLOCK + yield` 允许线程在无数据时
短暂让出 CPU，当对端 fd 最终关闭时（Arc 引用归零），splice 会返回错误从而退出循环。

如果未来需要更精确的半关闭处理，可以在一个方向 EOF 后，对另一个方向的 socket 调用
`shutdown(Write)` 来主动通知。但这需要额外的跨线程通信机制，复杂度较高。

### Issue 2: 限流器行为与预期不符

**文件**: `src/limiter.rs`

**问题描述**:

`Semaphore::acquire_owned().await` 在所有 permit 被占用时会**阻塞等待**，只有 Semaphore 被 `close()` 时才返回 `Err`。因此 `ConnectionLimitExceeded` 错误永远不会触发。当前行为是：超限连接排队等待，而非快速拒绝。

**修复方案**:

根据期望的行为选择：

- **快速拒绝（推荐）**：使用 `try_acquire_owned()`，超限立即返回错误并关闭连接：

```rust
pub fn try_acquire(&self) -> Result<OwnedSemaphorePermit, SocksError> {
    match self.semaphore.clone().try_acquire_owned() {
        Ok(permit) => Ok(permit),
        Err(_) => Err(SocksError::ConnectionLimitExceeded(self.max_connections)),
    }
}
```

- **排队等待（当前行为）**：如果确实期望排队，则应移除 `ConnectionLimitExceeded` 错误变体，避免误导，并在文档中说明行为。

建议同时在 `server.rs` 的 `handle_connection` 中，限流拒绝时向客户端发送 SOCKS5 错误响应（`REP_GENERAL_FAILURE`），而非直接断开。

## P1：性能优化

### Issue 3: 缺少 TCP_NODELAY

**文件**: `src/server.rs`

**问题描述**:

SOCKS5 握手阶段有多次小包交互（2 字节版本+方法数、2 字节方法选择响应、请求、响应），Nagle 算法会将这些小包合并，引入最高 40ms 的延迟。

**修复方案**:

在 accept 后立即设置 `TCP_NODELAY`，对目标连接也同样设置：

```rust
// accept 后
socket.set_nodelay(true)?;

// 连接目标后
target.set_nodelay(true)?;
```

### Issue 4: Address::to_host() 产生不必要的堆分配

**文件**: `src/socks5/protocol.rs`, `src/server.rs`

**问题描述**:

`Address::to_host()` 返回 `String`，对于 IPv4/IPv6 地址会做一次格式化字符串分配，然后 `TcpStream::connect((string, port))` 又会重新解析这个字符串。这是一次完全不必要的 `format → parse` 往返。

**修复方案**:

为 `Address` 实现直接转换为 `SocketAddr` 的方法，对于域名走 `ToSocketAddrs`：

```rust
impl Address {
    pub async fn connect(&self, port: u16) -> Result<TcpStream, SocksError> {
        match self {
            Address::IPv4(addr) => {
                let sock_addr = SocketAddr::new(IpAddr::V4(*addr), port);
                Ok(TcpStream::connect(sock_addr).await?)
            }
            Address::IPv6(addr) => {
                let sock_addr = SocketAddr::new(IpAddr::V6(*addr), port);
                Ok(TcpStream::connect(sock_addr).await?)
            }
            Address::Domain(domain) => {
                Ok(TcpStream::connect((domain.as_str(), port)).await?)
            }
        }
    }
}
```

在 `server.rs` 中直接调用 `request.address.connect(request.port).await?`。

### Issue 5: 握手阶段 methods 使用 Vec 堆分配

**文件**: `src/socks5/handshake.rs`

**问题描述**:

每次握手都 `vec![0u8; nmethods]` 做堆分配。RFC1928 规定 `NMETHODS` 最大 255，实际通常 1-3 个。

**修复方案**:

使用栈上固定大小数组：

```rust
let mut methods = [0u8; 255];
stream.read_exact(&mut methods[..nmethods]).await?;

let supported = methods[..nmethods].contains(&AUTH_NO_AUTH);
```

## P2：健壮性与可用性

### Issue 6: 连接目标无超时控制

**文件**: `src/server.rs`

**问题描述**:

`TcpStream::connect()` 没有超时，如果目标主机不可达但未拒绝连接（如防火墙丢包），将等待系统默认的 TCP 超时（通常 2 分钟），期间占用一个连接 permit。

**修复方案**:

使用 `tokio::time::timeout` 包裹连接操作：

```rust
use tokio::time::{timeout, Duration};

let connect_timeout = Duration::from_secs(config.connect_timeout); // 建议默认 10s
let target = timeout(connect_timeout, TcpStream::connect(...))
    .await
    .map_err(|_| SocksError::ConnectTimeout)?
    .map_err(|e| SocksError::ConnectFailed(e.to_string()))?;
```

需要在 `AppConfig` 中增加 `connect_timeout` 配置项，在 `SocksError` 中增加 `ConnectTimeout` 变体。

### Issue 7: 缺少优雅关闭（Graceful Shutdown）

**文件**: `src/server.rs`, `src/main.rs`

**问题描述**:

`server::run` 是无限 `loop`，没有信号处理。进程被杀时所有活跃连接直接断开，可能导致客户端异常。

**修复方案**:

使用 `tokio::signal` 监听 SIGTERM/SIGINT，配合 `CancellationToken` 或 `broadcast` 通知所有连接优雅退出：

```rust
use tokio::signal;
use tokio_util::sync::CancellationToken;

pub async fn run(config: AppConfig) -> Result<(), SocksError> {
    let listener = TcpListener::bind(config.bind).await?;
    let cancel_token = CancellationToken::new();

    loop {
        tokio::select! {
            result = listener.accept() => {
                let (socket, peer_addr) = result?;
                let token = cancel_token.child_token();
                tokio::spawn(async move {
                    tokio::select! {
                        result = handle_connection(socket, peer_addr, limiter) => { ... }
                        _ = token.cancelled() => { info!("Connection cancelled"); }
                    }
                });
            }
            _ = signal::ctrl_c() => {
                info!("Shutdown signal received");
                cancel_token.cancel();
                break;
            }
        }
    }
    Ok(())
}
```

需要添加依赖 `tokio-util`。

### Issue 8: 使用 tracing Span 替代逐步日志

**文件**: `src/server.rs`

**问题描述**:

当前每个连接的每个阶段都有独立的 `debug!` 日志调用，高并发下日志格式化本身有开销，且日志之间缺乏关联性。

**修复方案**:

使用 `tracing::instrument` 或手动创建 span，将连接的所有日志关联到同一个 span：

```rust
use tracing::{info_span, Instrument};

tokio::spawn(
    handle_connection(socket, peer_addr, limiter)
        .instrument(info_span!("conn", %peer_addr))
);
```

在 `handle_connection` 内部减少日志点，只保留关键节点（连接建立、连接关闭、错误）。

### Issue 9: 域名解析使用 from_utf8_lossy

**文件**: `src/socks5/request.rs`

**问题描述**:

`String::from_utf8_lossy` 会将非 UTF-8 字节静默替换为 `�`，导致域名被篡改但不报错。

**修复方案**:

```rust
let domain = String::from_utf8(domain_buf)
    .map_err(|e| SocksError::InvalidAddress(
        format!("Invalid domain encoding: {}", e)
    ))?;
```

### Issue 10: build_reply 函数位置不当

**文件**: `src/socks5/protocol.rs`, `src/socks5/reply.rs`

**问题描述**:

`build_reply` 是业务逻辑函数，放在 `protocol.rs`（类型定义文件）中职责不清。

**修复方案**:

将 `build_reply` 从 `protocol.rs` 移动到 `reply.rs` 中，`protocol.rs` 只保留常量和类型定义。同时更新 `request.rs` 中对 `build_reply` 的引用路径。

## P3：进阶优化（远期）

### Issue 11: 考虑 io_uring 替代 splice

在 Linux 5.6+ 内核上，`io_uring` 可以实现真正的异步零拷贝，避免阻塞线程池的开销。可关注 `tokio-uring` 的成熟度，待其稳定后迁移。

### Issue 12: SO_REUSEPORT 多监听器

在多核服务器上，使用 `SO_REUSEPORT` 创建多个监听 socket，每个绑定到独立的 Tokio runtime，避免 accept 成为瓶颈。

### Issue 13: 方案文档勘误

方案中"关键性能优化点"表格里的"连接池：Tokio 内置 TCP 连接复用"描述不准确。Tokio 的 `TcpStream` 没有内置连接池，每次 `connect()` 都是新建连接。SOCKS5 代理的 CONNECT 模式本身也不适合连接池。建议删除或修正该条目。
