# 移除 splice 实现，统一使用优化后的 tokio 异步转发

## 背景

`src/relay.rs` 原先采用平台分支策略：
- **Linux**：使用 `splice(2)` 系统调用实现零拷贝转发（`src_socket → pipe → dst_socket`）
- **非 Linux**：使用 `tokio::io::copy_bidirectional` 作为 fallback

零拷贝听起来是极致性能优化，但经过深入分析，在 SOCKS5 代理的典型负载下，splice 实现不仅没有带来大幅性能提升，反而引入了严重的并发瓶颈。

---

## 为什么移除 splice

### 1. 阻塞线程池瓶颈 — 并发能力被严重限制

splice 是阻塞系统调用，必须在 `tokio::task::spawn_blocking` 中执行。每个连接的每个方向占用一个阻塞线程：

```
每个连接 = 2 个阻塞线程（client→target + target→client）
tokio 默认阻塞线程池 = 512 个线程
最大并发连接数 = 512 / 2 = 256 个
```

而 `tokio::io::copy` 是纯异步的，基于 epoll/kqueue 事件驱动，**一个 tokio worker 线程可以同时处理成千上万个连接**。

> **结论**：splice 在并发连接数维度上，性能远劣于异步方案。

### 2. 半关闭（half-close）处理困难

当一个方向 EOF 时，需要唤醒另一个方向阻塞在 splice 上的线程。原实现使用 `shutdown(Read)` 来触发唤醒，但：

- 在某些内核版本上 `shutdown(Read)` 不一定能唤醒阻塞的 splice
- 如果唤醒失败，阻塞线程会挂起直到 TCP keepalive 超时（通常 2 小时），白白占用线程池资源
- 异步方案通过 `AsyncWriteExt::shutdown()` 天然支持半关闭，无需额外处理

### 3. 每连接额外的系统资源开销

| 资源 | splice 方案 | 异步方案 |
|------|------------|---------|
| 额外 fd | 4 个（2 个 pipe × 2 方向） | 0 个 |
| 每连接系统调用 | `pipe()` × 2 创建 + `close()` × 4 销毁 | 无额外开销 |
| 线程栈内存 | ~8 MB（2 线程 × 4 MB 默认栈） | 0（异步 task 仅几百字节） |

在高频短连接场景下，pipe 的创建/销毁开销不可忽视。

### 4. 代码复杂度与维护成本

- splice 实现涉及 `unsafe` 代码（`BorrowedFd::borrow_raw`）、`Arc` 生命周期管理、跨线程 shutdown 协调
- 需要 `nix` crate 作为 Linux-only 依赖，增加编译时间
- `#[cfg(target_os = "linux")]` 条件编译导致非 Linux 开发者无法测试 splice 路径
- 两套实现意味着两套需要维护和测试的代码

### 5. 零拷贝优势在 SOCKS5 场景下不显著

splice 的核心优势是避免 `kernel → userspace → kernel` 的两次内存拷贝。但：

- SOCKS5 代理的典型负载是**大量并发的小数据包**（HTTP 请求、API 调用），单次拷贝量小
- 现代 CPU 的内存带宽远超网络带宽，64 KiB 的内存拷贝耗时约 **1-2 μs**，而网络 RTT 通常在 **毫秒级**
- 内存拷贝的开销被网络延迟完全掩盖

### 量化对比总结

| 维度 | splice（原方案） | 异步转发（新方案） |
|------|-----------------|-------------------|
| 数据拷贝 | 0 次 | 2 次（可忽略） |
| 并发模型 | 每连接 2 个阻塞线程 | 纯异步，epoll/kqueue 驱动 |
| 最大并发 | ~256 连接 | 数万连接 |
| 每连接额外 fd | 4 个 | 0 个 |
| 每连接内存 | ~8 MB 线程栈 | ~几百字节 task 状态 |
| 半关闭处理 | 复杂，有边界问题 | 天然支持 |
| unsafe 代码 | 有 | 无 |
| 平台依赖 | Linux only | 全平台 |

## 优化后的异步转发方案

### 核心设计

移除所有平台分支代码，统一使用一套优化后的异步实现：

```
client_socket ──read──→ [64KB buffer] ──write──→ target_socket  (方向1)
target_socket ──read──→ [64KB buffer] ──write──→ client_socket  (方向2)
```

### 优化点 1：缓冲区从 8 KiB 提升到 64 KiB

`tokio::io::copy` 和 `copy_bidirectional` 内部默认使用 **8 KiB** 缓冲区。对于代理转发场景，这意味着每 8 KB 数据就需要一次 `read` + `write` 系统调用对。

将缓冲区提升到 **64 KiB**：
- 与 Linux 默认 pipe 容量对齐
- 与常见 TCP 窗口大小匹配
- 系统调用次数减少 **8 倍**
- 在大数据传输场景下吞吐量提升显著

实现方式：使用 `BufReader::with_capacity(64KB)` + `tokio::io::copy_buf`，而非修改 tokio 内部默认值。

### 优化点 2：split + spawn 替代 copy_bidirectional

原 fallback 使用 `copy_bidirectional(&mut client, &mut target)`，它内部通过 `poll` 在同一个 task 中交替处理两个方向。

新方案使用 `tokio::io::split` 将 TcpStream 拆分为独立的读写半，每个方向独立 `tokio::spawn`：

```rust
let (client_reader, client_writer) = tokio::io::split(client);
let (target_reader, target_writer) = tokio::io::split(target);

// 两个方向完全独立并发
tokio::spawn(copy_with_shutdown(client_reader, target_writer));
tokio::spawn(copy_with_shutdown(target_reader, client_writer));
```

优势：
- 两个方向可以被调度到不同的 worker 线程，充分利用多核
- 每个方向的 `copy_buf` 可以独立使用完整的 64 KB 缓冲区
- 代码结构更清晰，每个方向的生命周期独立管理

### 优化点 3：主动 shutdown 实现干净的半关闭

每个方向拷贝完成（EOF）后，立即对 writer 执行 `shutdown()`：

```rust
async fn copy_with_shutdown(reader, writer) -> io::Result<u64> {
    let bytes = copy_buf(&mut BufReader::with_capacity(64KB, reader), writer).await?;
    writer.shutdown().await?;  // 通知对端 EOF
    Ok(bytes)
}
```

这确保：
- 当 client 关闭发送端时，target 能及时收到 FIN
- 当 target 关闭发送端时，client 能及时收到 FIN
- 不依赖 drop 的隐式行为，半关闭语义明确

## 变更清单

### `src/relay.rs`

- **删除** `linux_splice` 模块（~120 行），包括 `splice_loop`、`relay_splice`、所有 `unsafe` 代码
- **删除** `generic_relay` 模块和 `#[cfg]` 条件编译分支
- **新增** `RELAY_BUFFER_SIZE` 常量（64 KiB）
- **新增** `copy_with_shutdown` 泛型异步函数
- **重写** `relay` 公开接口，使用 split + spawn + copy_buf

### `Cargo.toml`

- **删除** `[target.'cfg(target_os = "linux")'.dependencies]` 下的 `nix` 依赖

### 不受影响

- `server.rs` — 调用 `relay::relay(socket, target)` 的签名和返回值不变
- 所有 relay 测试 — 接口不变，6 个测试全部通过

## 远期优化方向

如果未来需要在 Linux 上追求极致单连接吞吐（如大文件传输场景），可考虑：

1. **io_uring 方案**：`IORING_OP_SPLICE` 是真正异步的零拷贝，不需要阻塞线程。需要 Linux 5.6+ 内核，待 `tokio-uring` 生态成熟后可评估引入
2. **混合策略**：根据连接的数据量动态选择路径 — 小数据量用异步拷贝（高并发友好），大数据量切换到 io_uring splice（高吞吐友好）
