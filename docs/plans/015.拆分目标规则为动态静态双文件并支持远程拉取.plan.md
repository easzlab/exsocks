将 target-rules.yaml 拆分为 dynamic-target-rules.yaml（用户自定义，高优先级）和 static-target-rules.yaml（外部接口定期拉取，低优先级），支持从远程 ACL 接口定期拉取规则数据并自动生成静态规则文件。

# 拆分目标规则为动态/静态双文件并支持远程拉取

将单一的 `target-rules.yaml` 配置文件拆分为两份独立文件，分别由用户手动配置和代理服务器自动生成，同时支持从外部 HTTP 接口定期拉取 ACL 数据并转换为静态规则。

## User Review Required

> [!IMPORTANT]
> **优先级设计**：`dynamic-target-rules.yaml` 的所有规则整体优先级高于 `static-target-rules.yaml`。合并时 dynamic 规则排在前面，仍采用 min-priority-wins 语义。用户自定义规则始终优先于远程拉取的规则。

> [!NOTE]
> **文件不存在容错**：两个规则文件都允许不存在，不存在时视为空规则集。这使得 static 文件在首次拉取前不需要预先创建，dynamic 文件也可以选择性使用。

> [!IMPORTANT]
> **远程拉取为可选功能**：通过 `static_target_rules_fetch_enabled` 控制是否启用，默认关闭。不启用时 static 文件仍支持手动编写和热加载。

> [!NOTE]
> **外部依赖**：远程 ACL 接口通信复用已有的 `hyper-util Client`（HTTP 客户端）和 `serde_json`（JSON 解析），无需引入额外依赖。

## Background

原有设计使用单一的 `target-rules.yaml` 文件管理所有目标地址规则。在实际使用场景中存在两个需求：

1. **用户自定义规则**：用户需要手动配置自己的放行/阻止规则，这些规则应该具有最高优先级
2. **集中管控规则**：代理服务器需要定期从外部管控平台拉取统一的 ACL 策略，自动应用而无需人工干预

将规则文件拆分为 dynamic（用户自定义）和 static（自动生成）两份，可以清晰地分离关注点，避免自动更新覆盖用户的自定义配置。

## Proposed Changes

### 配置文件拆分设计

原有单一配置字段：

```yaml
target_rules_file: "target-rules.yaml"
```

替换为双文件 + 远程拉取配置：

```yaml
# 动态规则（用户自定义，高优先级）
dynamic_target_rules_file: "dynamic-target-rules.yaml"

# 静态规则（外部接口拉取，低优先级）
static_target_rules_file: "static-target-rules.yaml"

# 远程拉取配置（可选）
static_target_rules_fetch_enabled: false
static_target_rules_fetch_url: "http://127.0.0.1:8000/acl"
static_target_rules_fetch_interval: 36000  # 秒（10小时）
```

### 规则合并逻辑

```
┌─────────────────────────────────────┐
│         合并后的 TargetRuleSet       │
│                                     │
│  ┌─────────────────────────────┐    │
│  │ dynamic rules (优先级 0..N) │     │  ← 用户自定义，排在前面
│  └─────────────────────────────┘    │
│  ┌─────────────────────────────┐    │
│  │ static rules (优先级 N..M)  │     │  ← 远程拉取/自动生成，排在后面
│  └─────────────────────────────┘    │
│                                     │
│  min-priority-wins 语义不变          │
└─────────────────────────────────────┘
```

两份文件分别解析为规则列表后拼接：dynamic 规则在前（priority 值更小），static 规则在后。由于匹配引擎采用 min-priority-wins 语义，dynamic 规则将始终优先于 static 规则。

### 外部 ACL 接口数据转换

**接口响应格式**：

```json
{
  "data": {
    "netacl": {
      "BlockDomains": ["www.test.com", "login.foo.com", ...],
      "PassDomains": [".foo.com", ".bar.com", "api.cloudxx.com", ...],
      "BlockIPs": [],
      "PassIPs": ["44.11.3.8/32", "40.25.13.0/24", ...]
    }
  },
  "msg": "",
  "status": "ok"
}
```

**转换规则：**

| JSON 字段 | 条件 | 转换为 YAML 规则 |
|---|---|---|
| `PassDomains` | 以 `.` 开头（如 `.foo.com`） | `[DOMAIN-SUFFIX, foo.com, 0, 65535, PASS]` |
| `PassDomains` | 不以 `.` 开头（如 `api.cloudxx.com`） | `[DOMAIN, api.cloudxx.com, 0, 65535, PASS]` |
| `BlockDomains` | — | `[DOMAIN, www.bar.com, 0, 65535, BLOCK]` |
| `PassIPs` | — | `[IPCIDR, 44.11.3.8/32, 0, 65535, PASS]` |
| `BlockIPs` | — | `[IPCIDR, x.x.x.x/x, 0, 65535, BLOCK]` |

生成的 YAML 文件内写入顺序为：BlockDomains → PassDomains → BlockIPs → PassIPs。

### 定期拉取任务流程

```
┌──────────┐     ┌──────────────┐     ┌───────────────────┐
│ 定时触发  │────▶│ HTTP GET url │────▶│ 解析 JSON 响应     │
└──────────┘     └──────────────┘     └───────┬───────────┘
                                              │
                                    ┌─────────▼──────────┐
                                    │ convert_acl_to_yaml │
                                    └─────────┬──────────┘
                                              │
                                    ┌─────────▼──────────┐
                                    │ 写入 static 文件    │
                                    └─────────┬──────────┘
                                              │
                                    ┌─────────▼──────────┐
                                    │ file watcher reload│
                                    └────────────────────┘
```

- **首次启动立即拉取一次**，后续按 `fetch_interval` 定期拉取
- 拉取失败时保留上次的 static 文件内容，打印 warn 日志
- 写入成功后等待 file watcher 触发热加载
- 任务通过 `CancellationToken` 控制，服务器关闭时自动停止

**新增字段：**

```rust
/// 动态目标地址规则配置文件路径（用户自定义，高优先级）
#[serde(default = "default_dynamic_target_rules_file")]
pub dynamic_target_rules_file: PathBuf,

/// 静态目标地址规则配置文件路径（外部接口拉取，低优先级）
```

- `static_target_rules_fetch_url` → `""`
- `static_target_rules_fetch_interval` → `36000`

```rust
// 旧：单文件加载
```

```rust
        Err(e) if e.kind() == std::io::ErrorKind::NotFound => {
```

#### watch() 方法改造

```rust
// 旧：监听单个文件
pub fn watch(self: &Arc<Self>) -> Result<RecommendedWatcher, SocksError>

// 新：同时监听两个文件，返回多个 watcher
pub fn watch(self: &Arc<Self>) -> Result<Vec<RecommendedWatcher>, SocksError>
```

自动去重同目录的文件，对每个目录创建一个 watcher，监听目录内对应文件名的变更事件。任一文件变更时触发 reload（合并两份文件重新编译）。

#### 新增外部 ACL 接口相关

```rust
/// 外部 ACL 接口响应数据结构
#[derive(Deserialize)]
struct AclResponse {
    data: AclData,
    status: String,
}

#[derive(Deserialize)]
struct AclData {
    netacl: NetAcl,
}

#[derive(Deserialize)]
#[serde(rename_all = "PascalCase")]
struct NetAcl {
    block_domains: Vec<String>,
    pass_domains: Vec<String>,
    block_i_ps: Vec<String>,
    pass_i_ps: Vec<String>,
}

/// 将 ACL 数据转换为 target-rules YAML 格式
fn convert_acl_to_yaml(acl: &NetAcl) -> String

/// 从远程接口拉取 ACL 数据
async fn fetch_acl(url: &str) -> Result<String, SocksError>

/// 拉取并写入静态规则文件
async fn fetch_and_write_static(url: &str, path: &Path)

/// 启动定期拉取任务
pub fn start_fetch_task(
    self: &Arc<Self>,
    fetch_url: String,
    interval: Duration,
    cancel_token: CancellationToken,
)
```

### [MODIFY] server.rs

在 `run_with_listener` 中适配新的初始化逻辑：

```rust
// 旧
let trc = Arc::new(TargetRuleControl::load(&config.target_rules_file)?);
let _watcher = trc.watch()?;

// 新
let trc = Arc::new(TargetRuleControl::load(
    &config.dynamic_target_rules_file,
    &config.static_target_rules_file,
)?);
let _watchers = trc.watch()?;

// 启动定期拉取任务（如果启用）
if config.static_target_rules_fetch_enabled
    && !config.static_target_rules_fetch_url.is_empty()
{
    trc.start_fetch_task(
        config.static_target_rules_fetch_url.clone(),
        Duration::from_secs(config.static_target_rules_fetch_interval),
        cancel_token.clone(),
    );
}
```
