# IPCIDR 规则匹配 Radix Trie 优化

将 `target_rules.rs` 中 IPCIDR 规则的匹配数据结构从 `Vec<CidrEntry>` 线性遍历（O(K)，K=规则总数）替换为基于 `prefix-trie` crate 的 Radix Trie（O(W+M)，W=地址位宽 32/128，M=匹配前缀上的规则总数），利用 `cover_values` API 遍历所有匹配前缀并取最小 priority（min-priority-wins），实现高效查找。

> [!NOTE]
> **性能适用场景**：在典型小规模配置（< 50 条 CIDR 规则）下，Vec 线性扫描的缓存友好性与 Trie 方案性能差异可忽略。Trie 的优势在规则数量增长到数百条以上时体现，查询耗时不随规则总数 K 增长。

> [!WARNING]
> **同一 CIDR 不同端口范围的处理**：`PrefixMap` 的同一个 key 只能存一个 value。如果同一个 CIDR（如 `10.0.0.0/8`）配置了多条规则（不同端口范围），需要将 value 类型改为 `Vec<PrioritizedRule>`。因此 Trie 的类型应为 `PrefixMap<Ipv4Net, Vec<PrioritizedRule>>`。

**4. 修改 `match_cidr` 方法**

利用 `prefix-trie` 的 `cover_values` API 遍历所有覆盖目标 IP 的前缀节点，取 priority 最小且端口匹配的规则（min-priority-wins，即配置文件中最靠前的匹配规则优先）。

注意：与原 Vec 方案的 early return 不同，Trie 方案必须遍历所有匹配前缀才能确定全局最小 priority，因为 `cover_values` 的遍历顺序（从根到叶，即从宽到窄）不保证 priority 递增：

```rust
fn match_cidr(&self, ip: &IpAddr, port: u16) -> Option<&PrioritizedRule> {
    let normalized = normalize_ip(*ip);

    match normalized {
        IpAddr::V4(v4) => {
            let host_prefix = Ipv4Net::new(v4, 32).unwrap();
            let mut best: Option<&PrioritizedRule> = None;
            // cover_values 遍历所有覆盖 host_prefix 的前缀节点
            // 即所有 contains(v4) 为 true 的 CIDR 规则
            for rules in self.cidr_trie_v4.cover_values(&host_prefix) {
                for r in rules {
                    if r.port_matches(port) && r.is_higher_priority_than(best) {
                        best = Some(r);
                    }
                }
            }
            best
        }
        IpAddr::V6(v6) => {
            let host_prefix = Ipv6Net::new(v6, 128).unwrap();
            let mut best: Option<&PrioritizedRule> = None;
            for rules in self.cidr_trie_v6.cover_values(&host_prefix) {
                for r in rules {
                    if r.port_matches(port) && r.is_higher_priority_than(best) {
                        best = Some(r);
                    }
                }
            }
            best
        }
    }
}
```

> [!NOTE]
> **`cover_values` 语义**：给定一个 `/32`（或 `/128`）的主机前缀，`cover_values` 返回所有**包含**该主机地址的 CIDR 前缀的值。这等价于遍历 Trie 从根到叶路径上所有带值的节点。
>
> **复杂度分析**：Trie 遍历本身为 O(W)（W = 地址位宽，IPv4: 32, IPv6: 128），与规则总数 K 无关。但由于每个匹配前缀的 value 是 `Vec<PrioritizedRule>`，还需遍历所有匹配前缀上的规则进行端口匹配和优先级比较，因此实际复杂度为 **O(W + M)**，其中 M = 所有匹配前缀上的规则总数。在典型配置中，嵌套 CIDR 层级和同一 CIDR 的多端口规则都很少，M 通常 < 5，实际表现接近 O(W) 常数时间。相比原始 Vec 方案的 O(K)（K = 全部规则数），在规则数量较大时有显著优势。
>
> **`cover_values` 零堆分配**：经源码验证（`prefix-trie` 0.8.2），`cover_values` 返回的 `CoverValues` 迭代器仅包含 `table: &Table`（引用）、`idx: Option<usize>`（栈上 8 字节）、`prefix: &P`（引用）三个字段，完全栈分配。`next()` 方法通过 `left`/`right` 子节点索引遍历树，无任何堆分配，适合高频调用场景。
>
> **内存开销分析**：`PrefixMap` 底层为 `Vec<Node<P, T>>`，每个 `Node` 包含 `prefix: P`、`value: Option<T>`、`left/right: Option<NonZeroUsize>`。对于 IPv4（`Ipv4Net` = 5 字节），单个 `Node<Ipv4Net, Vec<PrioritizedRule>>` 约 48 字节。注意 Trie 可能产生不含值的分支节点（branching nodes），节点总数可能多于规则数。对于 K 条非重叠规则，节点数上限约 2K。在小规模配置（< 20 条规则）下，额外内存开销仅数 KB，完全可忽略。
