## 整体架构性能梳理

当前服务器处理一个连接的完整生命周期分为 **6 个阶段**：

```
Accept → 限流检查 → SOCKS5 握手 → 请求解析 → 连接目标 → 双向转发
```

---

### 阶段 1：连接接受（Accept）

**当前实现**：
- `tokio::select!` 监听 `listener.accept()` + `ctrl_c` + `cancel_token`
- 每个连接 `tokio::spawn` 独立任务
- 立即设置 `set_nodelay(true)` 禁用 Nagle 算法

**预期性能**：⭐⭐⭐⭐⭐ **优秀**
- Accept 本身是 O(1) 操作，epoll/kqueue 事件驱动
- `tokio::spawn` 创建异步 task 仅需几百字节内存，开销极低

**可优化点**：
- **`SO_REUSEPORT` 多监听器**（远期）：在多核服务器上，accept 可能成为单点瓶颈。使用 `SO_REUSEPORT` 创建多个监听 socket，每个绑定到独立 worker，可以提升 accept 吞吐。但在当前负载下不是瓶颈，属于远期优化。

---

### 阶段 2：限流检查

**当前实现**：
- 基于 `tokio::sync::Semaphore` 的 `try_acquire_owned()`
- 非阻塞 O(1) 操作，超限立即拒绝

**预期性能**：⭐⭐⭐⭐⭐ **优秀**
- 原子操作，无锁竞争（CAS）
- 每连接仅一个 `OwnedSemaphorePermit` 句柄，RAII 自动释放

**无需优化**：设计已经是最优方案。

---

### 阶段 3：SOCKS5 握手

**当前实现**：
- 栈上固定缓冲区：`[0u8; 2]` + `[0u8; 255]`
- 2 次 `read_exact` + 1 次 `write_all`
- 零堆分配

**预期性能**：⭐⭐⭐⭐⭐ **优秀**
- 全部栈上操作，无堆分配
- 系统调用次数已是协议要求的最小值（2 读 + 1 写）

**可优化点**：
- ⚠️ **合并读写减少系统调用**：理论上可以将版本+方法数（2 字节）和方法列表（N 字节）合并为一次 `read`，但 `read_exact` 的语义要求精确字节数，且第二次读取的长度依赖第一次的结果，**无法合并**。当前已是最优。

---

### 阶段 4：请求解析

**当前实现**：
- 栈上缓冲区：`[0u8; 4]`（请求头）、`[0u8; 4]`/`[0u8; 16]`（地址）、`[0u8; 2]`（端口）
- 域名场景：`vec![0u8; len]` 堆分配 + `String::from_utf8` 转换
- IPv4：3 次 `read_exact`；域名：4 次 `read_exact`；IPv6：3 次 `read_exact`

**预期性能**：⭐⭐⭐⭐ **良好**

**可优化点**：

| 优化项 | 当前 | 优化后 | 收益 |
|--------|------|--------|------|
| **域名缓冲区** | `vec![0u8; len]` 堆分配 | `[0u8; 255]` 栈上固定数组 | 消除每个域名连接的一次堆分配 |
| **合并读取** | 请求头 4 字节单独读 | 可以用更大的栈缓冲区一次读取请求头+地址 | 减少 1 次系统调用 |
| **IPv6 解析** | 手动 8 次 `u16::from_be_bytes` | `Ipv6Addr::from(<[u8; 16]>)` 直接构造 | 代码简化，性能相当 |

> **域名栈上缓冲区优化**是最有价值的改动：RFC1928 规定域名最长 255 字节，完全可以用栈数组替代 Vec。

---

### 阶段 5：连接目标

**当前实现**：
- `tokio::time::timeout(connect_timeout, request.address.connect(port))`
- `Address::connect()` 对 IPv4/IPv6 直接构造 `SocketAddr`，避免字符串格式化
- 默认超时 10 秒

**预期性能**：⭐⭐⭐⭐ **良好**
- 连接延迟主要取决于网络 RTT，代码层面开销可忽略

**可优化点**：
- ⚠️ **DNS 解析缓存**：域名场景下，`TcpStream::connect((domain, port))` 每次都会触发 DNS 解析。对于高频访问的域名，可以引入一个简单的 TTL DNS 缓存（如 `HashMap<String, (Vec<SocketAddr>, Instant)>`），避免重复解析。
- ⚠️ **Happy Eyeballs (RFC 6555)**：当域名同时有 A 和 AAAA 记录时，当前实现是串行尝试。实现 Happy Eyeballs 算法可以并行尝试 IPv4/IPv6，选择最先成功的，减少连接延迟。

---

### 阶段 6：双向数据转发（核心热路径）

**当前实现**：
- `BufReader::with_capacity(64 KiB)` + `tokio::io::copy_buf`
- `tokio::io::split` 拆分读写半，每方向独立 `tokio::spawn`
- 主动 `shutdown()` 实现半关闭
- `CancellationToken` 支持优雅关闭
- 每连接内存：~128 KiB（双向缓冲区）

**预期性能**：⭐⭐⭐⭐ **良好**

**可优化点**：

| 优化项 | 描述 | 预期收益 |
|--------|------|----------|
| **`split` 的 Mutex 开销** | `tokio::io::split` 内部使用 `Arc<Mutex>` 保护底层 socket，每次 read/write 都要获取锁 | 改用 `TcpStream::into_split()` 可以获得零开销的 `OwnedReadHalf`/`OwnedWriteHalf`，**消除所有锁竞争** |
| **缓冲区复用** | 当前每个连接分配 2 × 64 KiB 缓冲区，连接结束后释放 | 引入对象池（如 `crossbeam` 的 `ArrayQueue`）复用缓冲区，减少高频短连接场景下的分配/释放开销 |
| **io_uring**（远期） | Linux 5.6+ 的 `IORING_OP_SPLICE` 是真正异步的零拷贝 | 待 `tokio-uring` 成熟后可评估，兼顾零拷贝和高并发 |

> ⚡ **`split` → `into_split` 是当前最有价值的优化**：`tokio::io::split` 使用通用的 `Arc<Mutex<Inner>>` 实现，而 `TcpStream::into_split()` 利用 TCP socket 的全双工特性，返回的 `OwnedReadHalf`/`OwnedWriteHalf` 完全无锁。在高吞吐场景下，每次 read/write 省去一次 mutex lock/unlock，累积效果显著。

---

### 阶段 7：回复构建

**当前实现**：
- `build_reply()` 使用 `Vec<u8>` 堆分配构建响应
- `Address::to_bytes()` 也返回 `Vec<u8>`
- 一次 `write_all` 发送

**预期性能**：⭐⭐⭐ **中等**（但影响极小，每连接仅执行一次）

**可优化点**：
- ⚠️ **栈上构建回复**：SOCKS5 回复最大长度 = 3（头）+ 1（ATYP）+ 16（IPv6）+ 2（端口）= **22 字节**。完全可以用 `[u8; 22]` 栈数组 + 实际长度计数器替代 `Vec<u8>`，消除 2 次堆分配（`build_reply` + `to_bytes`）。
